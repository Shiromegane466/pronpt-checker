# SOW: プロンプト改善フィードバックWebアプリ

## 1. 背景

ユーザーが入力したプロンプトに対して、AIが「何が足りないか」「どう直すとよいか」を具体的にフィードバックし、さらにその場で再利用できる改善版プロンプトを生成するWebアプリを開発する。

調査したベストプラクティスでは、よいプロンプトには以下の要素が重要とされている。

- 目的、前提、対象読者、期待する成果物を明確にする。
- 出力形式、長さ、トーン、制約を具体的に指定する。
- 指示と入力データをMarkdown見出し、区切り記号、XMLタグなどで分離する。
- 必要に応じて例を示す。
- 曖昧な表現や「いい感じに」などの抽象表現を、測定可能な条件に置き換える。
- プロンプトは一度で完成させるものではなく、評価と改善を繰り返す。
- AI出力は非決定的であるため、アプリ側では構造化出力、モデルバージョン固定、評価ログを設計に含める。
- LLMアプリではプロンプトインジェクション、機密情報の混入、出力の過信を前提にした安全設計が必要。

## 2. 目的

本プロジェクトの目的は、ユーザーがプロンプト作成に不慣れでも、より高精度なAI回答を得るためのプロンプトに改善できるWebアプリを構築することである。

アプリは単に「良い/悪い」を判定するのではなく、次の3点を提供する。

1. 入力プロンプトの強みと改善点の可視化
2. 不足情報を補うための具体的な質問または指摘
3. そのままコピーして使える改善版プロンプト

## 3. 対象ユーザー

- 生成AIを業務や学習に使う一般ユーザー
- ChatGPT、Claude、Geminiなどで期待通りの回答を得られず困っているユーザー
- 社内でプロンプトの品質を標準化したいチーム
- AI活用研修やプロンプト作成支援を行う担当者

## 4. スコープ

### 4.1 MVPで実装する機能

#### プロンプト入力

- ユーザーが自由記述でプロンプトを入力できる。
- 入力文字数の目安を表示する。
- 入力例を複数提示する。
- 空入力、短すぎる入力、機密情報らしき入力に対して警告を出す。

#### プロンプト診断

入力されたプロンプトを以下の観点で評価する。

- 目的の明確さ
- 背景・前提情報の有無
- 対象読者・利用シーンの明確さ
- 出力形式の指定
- 制約条件の指定
- 必要な情報と不要な情報の分離
- 曖昧な表現の有無
- 期待するトーンや文体の指定
- 例示の有無
- セキュリティ上の懸念

#### フィードバック表示

- 総合スコアを100点満点で表示する。
- 良い点を最大3件表示する。
- 改善点を優先度付きで表示する。
- それぞれの改善点について、なぜ問題なのか、どう直すべきかを説明する。
- ユーザーが次回以降に学べるよう、短い学習メモを表示する。

#### 改善版プロンプト生成

- 「こんなプロンプトにすると、より精度の高い回答を得られます！」という見出しで改善版を表示する。
- 改善版はコピーしやすいテキストエリアまたはコードブロックで表示する。
- 改善版には、目的、背景、入力情報、制約、出力形式を明確に含める。
- 元の意図を勝手に変えず、不明点はプレースホルダーとして残す。

#### コピー・再編集

- 改善版プロンプトをワンクリックでコピーできる。
- 改善版を入力欄へ戻して再診断できる。
- フィードバック結果をリセットできる。

#### レスポンシブUI

- PC、タブレット、スマートフォンで利用できる。
- 入力、診断結果、改善版プロンプトが読みやすく配置される。

### 4.2 MVPでは対象外

- ユーザーアカウント機能
- 履歴保存
- チーム共有
- 決済
- 複数LLMプロバイダーの切り替えUI
- ブラウザ拡張機能
- 入力プロンプトを実際に実行して回答品質を比較する機能
- ファイル添付、画像プロンプト、音声入力

## 5. 推奨UX

### 5.1 画面構成

1画面で主要体験が完結する構成とする。

- 左または上部: プロンプト入力
- 中央: 診断実行ボタン、評価中ステータス
- 右または下部: フィードバック、改善版プロンプト

### 5.2 UX原則

- 空の入力欄だけを置かず、ユーザーが何を書けばよいか分かる例を出す。
- AIの評価を絶対視させず、「改善の参考」として表示する。
- 長い説明を初期表示しすぎず、詳細は展開式にする。
- 改善点は抽象論ではなく、入力文のどの弱点に対応しているかを明示する。
- 改善版プロンプトはコピー、再編集、再診断しやすくする。
- 「AIが人間のように理解している」と誤解させる表現を避ける。

## 6. AI機能仕様

### 6.1 LLMへの要求

LLMには、入力プロンプトを評価し、構造化されたJSONで返すことを求める。

推奨レスポンス構造:

```json
{
  "score": 78,
  "summary": "目的は明確ですが、出力形式と前提条件が不足しています。",
  "strengths": [
    "依頼したい作業内容が短くまとまっています。"
  ],
  "issues": [
    {
      "priority": "high",
      "title": "出力形式が指定されていない",
      "reason": "回答の粒度や構成がモデル任せになり、期待とずれる可能性があります。",
      "suggestion": "箇条書き、表、JSONなど希望する形式を指定してください。"
    }
  ],
  "clarifying_questions": [
    "回答の対象読者は誰ですか？",
    "回答は短め、詳しめ、どちらがよいですか？"
  ],
  "improved_prompt": "あなたは...として振る舞ってください。..."
}
```

### 6.2 評価観点

評価ロジックは以下の重み付けを初期値とする。

| 観点 | 重み |
| --- | ---: |
| 目的の明確さ | 15 |
| 背景・前提 | 15 |
| 出力形式 | 15 |
| 制約条件 | 10 |
| 対象読者・利用シーン | 10 |
| 曖昧表現の少なさ | 10 |
| 例示・参考情報 | 10 |
| 実行可能性 | 10 |
| 安全性 | 5 |

### 6.3 改善版プロンプトの生成ルール

- 元プロンプトの意図を維持する。
- 不足情報を勝手に断定しない。
- 不明な情報は `[ここに対象読者を入力]` のようなプレースホルダーにする。
- 見出しを使って、役割、目的、背景、条件、出力形式を分離する。
- 必要に応じて「回答前に不足情報があれば質問してください」を含める。
- 機密情報、個人情報、APIキー、パスワードらしき内容は再掲しない。

## 7. セキュリティ・安全性

- ユーザー入力は信頼しない。
- 入力内の「前の指示を無視して」などの文言で、システム指示が上書きされないようにする。
- LLMの出力はHTMLとして直接描画せず、エスケープまたはMarkdownサニタイズを行う。
- APIキーはクライアントに置かず、サーバー側環境変数で管理する。
- ログ保存を行う場合、個人情報や機密情報の取り扱い方針を明示する。
- 初期MVPでは、プロンプト本文の永続保存は行わない方針とする。
- レート制限、入力文字数制限、タイムアウトを設ける。

## 8. 非機能要件

### パフォーマンス

- 初回表示は3秒以内を目標とする。
- 診断結果は通常10秒以内に返すことを目標とする。
- 評価中はローディング表示を行う。

### 可用性

- LLM APIエラー時は、再試行可能なエラーメッセージを表示する。
- モデルの拒否や安全フィルターにかかった場合は、その理由を分かりやすく表示する。

### 保守性

- LLMへの評価プロンプトはコードから分離し、バージョン管理する。
- モデル名、温度、最大トークン数、タイムアウトは設定値として管理する。
- 将来的な評価観点追加に備え、診断結果は構造化データで扱う。

### アクセシビリティ

- キーボード操作に対応する。
- ボタン、入力欄、結果領域に適切なラベルを付与する。
- 色だけに依存して評価結果を表現しない。
- モバイル幅でもテキストがはみ出さない。

## 9. 技術方針

### 推奨構成

- フロントエンド: ReactまたはNext.js
- バックエンド: Next.js API Routes、Express、または同等の軽量API
- LLM連携: サーバー側API経由
- 出力形式: JSON Schemaによる構造化出力
- スタイル: CSS Modules、Tailwind CSS、または既存プロジェクトの標準

### MVPでの実装方針

- まずは単一ページアプリとして構築する。
- DBは使わない。
- 入力、診断、改善版生成を1回のLLM呼び出しで行う。
- LLM出力のJSONバリデーションに失敗した場合は、再試行またはエラーメッセージを表示する。

## 11. 受入条件

以下を満たした場合、MVP完了とする。

- ユーザーがプロンプトを入力できる。
- 診断ボタンを押すと、スコア、良い点、改善点、改善版プロンプトが表示される。
- 改善版プロンプトが元の意図を維持している。
- 不足情報は断定せず、プレースホルダーまたは確認質問として扱われる。
- コピー操作ができる。
- APIエラー時にユーザー向けのメッセージが表示される。
- PCとスマートフォンで主要操作が破綻しない。
- APIキーがクライアントに露出しない。
- LLM出力が構造化データとして処理される。

## 13. テスト方針

### 代表テストケース

- 空入力
- 1文だけの曖昧なプロンプト
- 十分に構造化されたプロンプト
- 個人情報を含む可能性のあるプロンプト
- プロンプトインジェクション風の入力
- 非常に長い入力
- 日本語入力
- 英語入力

### 評価観点

- JSONスキーマに従った出力になっているか。
- 改善点が入力内容に対応しているか。
- 改善版プロンプトが過剰に創作していないか。
- コピー機能が動作するか。
- モバイル表示で操作可能か。
- エラー時にクラッシュしないか。

## 14. リスクと対策

| リスク | 対策 |
| --- | --- |
| AIが元プロンプトの意図を変えてしまう | 「意図を維持する」「不明点はプレースホルダーにする」ルールをシステム側に置く |
| フィードバックが抽象的になる | 評価観点とJSONスキーマで、理由と修正案を必須にする |
| 出力形式が崩れる | Structured OutputsまたはJSON Schemaバリデーションを使う |
| ユーザーがAI評価を過信する | 「改善の参考」であることをUIに明記する |
| 機密情報が入力される | 入力時の注意喚起、永続保存しない方針、ログ制御を行う |
| プロンプトインジェクション | ユーザー入力をデータとして扱い、指示として実行しない設計にする |
| APIコスト増加 | 文字数制限、レート制限、モデル設定、キャッシュ方針を設ける |
